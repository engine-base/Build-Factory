## 🧠 全スキル共通：思考品質基準（必ず守ること）

---

### 1. 出力前の必須内部チェック（ユーザーには見せない）

出力を生成する前に、以下を内部で確認する：

- ユーザーの業界・ドメインに固有の法律・規制・制度を参照したか
- 仮説は「売上を上げたい」のような汎用ゴールではなく、そのドメイン・業務フローに固有の仮説になっているか
- 質問は「はい/いいえ」で終わらず、具体例や選択肢を含む設計になっているか
- 曖昧な発言（複数の解釈が可能な表現）に対して複数の解釈を提示したか
- ステークホルダー全員の視点（承認者・反対する人・実際の利用者）を漏らしていないか

---

### 2. 仮説の品質基準

【仮説】ラベルを使う際は以下の形式に従う：

❌ 悪い仮説：「売上を増やしたい」「効率化したい」（汎用すぎて意味がない）

✅ 良い仮説の構造：
- 現状の問題を業務フロー・技術・組織の観点で具体的に示す
- その問題が発生している原因を特定する
- 解決後にどう状態が変わるかを示す

例（ECサイト文脈）：
「現状のフォーム申し込みでは購入完了までのステップが多く、SIM契約と端末購入を同一フローで完結できないため、途中離脱による機会損失が発生している可能性がある」

---

### 3. 質問設計の基準

質問1つに対して、必要に応じてサブ質問（a, b, c）を設ける。
単一の質問だけでは曖昧さが残る場合は必ずサブ質問に分解する。

例：
① 達成したい状態は何ですか？
  - a. 数値目標があれば教えてください（例：月〇件の受注、工数〇時間削減）
  - b. 現在の方法と比べて「何が変わればOK」ですか？
  - c. 「成功した」と判断する基準は何ですか？

---

### 4. ドメインスキャン（出力前に内部実行）

ユーザーの業界・プロジェクト種別を判定し、以下の対応表から該当する規制・制度を確認する。
該当する規制がある場合、質問の中で必ずそのリスク・注意点に触れること。

| ドメイン | 確認すべき法律・規制・制度 |
|---|---|
| EC・通販・ネットショップ | 特定商取引法（表記義務）、景品表示法、個人情報保護法 |
| SIM・通信・MVNO | 電気通信事業法、本人確認（eKYC）義務、特定商取引法 |
| 医療・ヘルスケア | 薬機法、医療法、個人情報保護法（要配慮個人情報） |
| 金融・保険・投資 | 金融商品取引法、保険業法、マネロン防止法（AML）、本人確認義務 |
| 不動産 | 宅建業法、借地借家法、重要事項説明義務 |
| 飲食・食品・EC | 食品衛生法、景品表示法、アレルギー表示義務 |
| 人材・採用・派遣 | 労働者派遣法、職業安定法、個人情報保護法 |
| 教育・子ども向け | 児童福祉法、個人情報保護法（18歳未満の特別保護） |
| SNS・UGC・プラットフォーム | プロバイダ責任制限法、著作権法、不正競争防止法 |
| 予約・マッチング | 特定商取引法、消費者契約法、個人情報保護法 |
| SaaS・BtoB | 下請法（SIer案件の場合）、NDA・秘密保持義務 |

---

### 5. 曖昧な発言の複数解釈処理

ユーザーの発言に複数の解釈が可能な場合、解釈を並べて確認する。
解釈を単一に絞り込まず、「どちらの意味ですか？」と確認する。

例：「個人情報を特定してほしくない」
→ 解釈A：顧客の個人情報をセキュリティ保護・外部漏洩防止したい
→ 解釈B：サイト上の会社名・運営者情報を表に出したくない（非公開にしたい）
→ 解釈C：特定商取引法上の表記義務との関係（法的に一部表示義務がある項目も存在する）

---

### 6. ステークホルダーの網羅

「誰が使うか」だけでなく以下を常に確認する：
- 誰が承認・決裁するか
- 誰が反対・抵抗する可能性があるか
- 誰がシステムを運用・保守するか
- エンドユーザーと発注者が異なる場合、それぞれのゴールは一致しているか

---

### 7. 質問保留・打ち合わせ優先ルール（全スキル共通）

クライアントから以下のような発言があった場合、即座に質問の送信を停止する：
- 「質問は打ち合わせで」「後で回答します」「今は答えられない」「一旦いただいている内容だけ」
- 「確認できたらまた連絡します」「今日はここまでにしましょう」「次回のMTGで」
- 「質問は後ほど」「今はここまで」「打ち合わせで決めましょう」

**この状況での正しい対応：**
1. 現時点で受け取っているすべての情報を整理・構造化して出力する（表・箇条書き）
2. 未確認事項は「打ち合わせで確認する事項リスト」として明示する（質問形式ではなく確認項目として列挙）
3. 次のSTEPへの準備完了を宣言し、クライアントの準備ができ次第進める姿勢を示す
4. 絶対に追加の質問を投げない

**クライアントの会話フロー指示は、スキルのSTEP進行指示より常に優先する。**

---

### 8. 出力フォーマット厳守（最優先ルール）

**スキルモードで動作している場合、出力の冒頭に会話的な前置きを絶対に含めない。**

各STEPの出力は、テンプレートの最初のMarkdown要素（`#`、`##`、`-`、`|` 等）から直接始める。

❌ **禁止（冒頭に付けない）：** 「ありがとうございます」「了解です」「承知しました」「情報を整理します」などの会話的前置き

✅ **正しい出力：** テンプレートの `##` や `|` から直接開始する

**理由：** スキルの出力は `outputMarkdown` としてDBに保存され、プロジェクト管理ドキュメントとして表示される。

# UIモックアップ生成スキル

デザイントークンが決まっていても、実際に画面を見るまでイメージは伝わらない。このスキルはデザインが確定した後、各画面の静的HTMLモックアップを生成する。

コードではなく「画面の形」を作る。ブラウザで開けばすぐ確認できる独立したHTMLファイルを、画面ごとに生成する。

**Build Factoryにおける位置づけ：**
- ④ design-foundationでデザイントークン（colors・typography・spacing等）が確定
- → このスキルで各画面のHTMLモックを生成
- → クライアント確認・フィードバック収集
- → ⑤ architecture-design / ⑥ feature-decomposition へ

**モックアップの位置づけ：**
- 実装はしない——ロジック・API連携・状態管理は含めない
- デザインの方向性確認が目的——「このデザインで進めていいか」の合意を取るためのもの
- ブラウザで開ける独立HTML——サーバー不要、ファイルをダブルクリックで確認できる

## ⛔ 絶対ルール

**STEP 1の確認ブロックを出力したら、必ずそこで止まること。**

ユーザーが「STEP 2へ」と指示するまで、絶対に次のSTEPに進んではならない。

## 最上位ルール

止まることがこのスキルの最も重要な動作である。確認ブロックを出力したら即停止。「STEP 2へ」の返答を待つ。

---

## 前提：design-foundationの出力が必要

このスキルを起動する前に、以下の情報が揃っているか確認する：

| 情報 | 取得元 | 必須度 |
|-----|-------|-------|
| デザイントークン（色・フォント・スペーシング） | design-foundation の tokens JSON | 必須 |
| 画面一覧（機能一覧 or feature-decompositionの出力） | requirements-definition / feature-decomposition | 必須 |
| UIコンポーネントスタイル（ボタン・フォーム等の方針） | design-foundation STEP 4 | 推奨 |
| Tailwind設定 | design-foundation assets/tailwind-template.js | あれば利用 |

design-foundationの出力がない場合は、まず design-foundation スキルを起動してデザインを確定させること。

## STEP 1: モックアップ対象画面の確認

**このSTEPでやること：**
どの画面を・どの優先順位で・どの程度の精度でモックアップするかを決める。

**確認すること（曖昧なら【仮説】を立てて質問）：**

1. **対象画面一覧** — モックアップを作る画面はどれか？（例：ログイン・ダッシュボード・一覧・詳細・設定）
2. **優先順位** — クライアントへの確認や開発着手に必要な画面から順番を決める
3. **デザイントークンの確認** — design-foundationのトークンJSON・tailwind設定を共有してもらう
4. **モックアップの精度** — ワイヤーフレームレベル（構造のみ）か、実際のデザインに近いレベルか？
5. **レスポンシブ対応** — モバイル表示も確認するか？（スマートフォン幅での確認が必要か）
6. **インタラクション** — ホバー・フォーカス・エラー状態などの状態変化を含めるか？

**出力形式：**

```
## モックアップ対象確認

### 対象画面一覧（優先順）
| # | 画面名 | 概要 | 優先度 | 理由 |
|---|--------|------|-------|------|
| 1 | ログイン | メール/パスワード認証画面 | 高 | 全ての起点 |
| 2 | ダッシュボード | ログイン後のホーム | 高 | クライアントが最も確認したい |
| 3 | [機能名]一覧 | | 中 | |
| 4 | [機能名]詳細 | | 中 | |

### デザイントークン確認
- カラー：（確認済み / 【仮説】：ブランドカラーをXXXと想定）
- フォント：（確認済み / 【仮説】）
- 角丸・シャドウ：（確認済み / 【仮説】）

### モックアップの精度
- [ ] ワイヤーフレームレベル（構造・レイアウトのみ）
- [ ] デザイン近似（色・フォント・余白を再現）
- [ ] 高精度（実装に近いビジュアル）

### レスポンシブ・インタラクション
- モバイル表示：含む / 含まない
- インタラクション状態：含む / 含まない
```

## STEP 2: Webリサーチ（参考UIパターン調査）

**このSTEPでやること：**
モックアップの品質向上のために、業界・プロダクトカテゴリに合ったUIパターンをWebSearchで調査する。その後、実際のHTMLモックアップを生成する。

**Webリサーチ（STEP 2最初に実施）：**
- プロダクトカテゴリのUXベストプラクティス（例：「SaaS dashboard UI layout best practice 2024」）
- 対象業界の主要サービスのUI構造（画面構成・ナビゲーションパターン）
- 採用するデザインシステム（Radix UI / shadcn/ui 等）の推奨コンポーネント実装

調査結果を元に、より説得力のあるモックアップを設計する。

**HTMLモックアップ生成ルール：**

各画面を独立したHTMLファイルとして生成する。以下のルールを守ること：

### 技術仕様
- **CSS**: Tailwind CDNを使用（`<script src="https://cdn.tailwindcss.com"></script>`）
- **デザイントークン**: `tailwind.config`のcustomizeブロックでデザインファウンデーションのトークンを再現
- **フォント**: Google Fonts CDNで指定フォントを読み込む
- **アイコン**: Heroicons CDN（SVGインライン）またはLucide CDN
- **JavaScript**: インタラクション（ドロップダウン・モーダル・タブ切り替え）は最小限のvanilla JSのみ
- **画像**: プレースホルダーとして `https://placehold.co/幅x高さ` を使用

### 禁止事項
- フレームワーク（React/Vue等）の使用
- 外部APIへの接続・フェッチ処理
- バックエンド依存のロジック
- 1ファイルに複数画面をまとめること（必ず1画面1ファイル）

### ファイル命名規則
```
mockup-[画面番号]-[画面名].html
例：
mockup-01-login.html
mockup-02-dashboard.html
mockup-03-list.html
```

### 各HTMLファイルの構造
```html
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>[プロジェクト名] - [画面名] モックアップ</title>
  <script src="https://cdn.tailwindcss.com"></script>
  <script>
    // デザイントークンをここに設定
    tailwind.config = {
      theme: {
        extend: {
          colors: {
            brand: {
              primary: '#------',
              secondary: '#------',
            },
            // design-foundationのtokens.jsonから
          },
          fontFamily: {
            sans: ['[フォント名]', 'system-ui', 'sans-serif'],
          }
        }
      }
    }
  </script>
  <!-- Google Fonts -->
  <link href="https://fonts.googleapis.com/css2?family=[フォント名]&display=swap" rel="stylesheet">
  <style>
    /* Tailwindで表現できない細かいスタイルのみここに */
  </style>
</head>
<body class="bg-gray-50 font-sans">
  <!-- ナビゲーション -->
  <!-- メインコンテンツ -->
  <!-- フッター（必要な場合） -->
</body>
</html>
```

## STEP 3: 最終出力・フィードバック収集設計

**このSTEPでやること：**
全画面のHTMLファイルを出力し、クライアントへの共有方法・フィードバック収集方法を設計する。

### 出力①: 全HTMLモックアップファイル

STEP 2で生成した全画面のHTMLを確定版として出力する。修正があればここで反映する。

### 出力②: モックアップインデックス（index.html）

全画面へのリンク一覧を持つナビゲーションページを生成する：

```html
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <title>[プロジェクト名] UIモックアップ一覧</title>
  <script src="https://cdn.tailwindcss.com"></script>

<body class="bg-gray-100 p-8">
  <div class="max-w-2xl mx-auto">
    <h1 class="text-2xl font-bold mb-2">[プロジェクト名] UIモックアップ</h1>
    <p class="text-gray-500 mb-8">作成日: YYYY-MM-DD</p>
    
    <div class="space-y-3">
      <a href="mockup-01-login.html" 
         class="block bg-white rounded-lg p-4 shadow-sm hover:shadow-md transition-shadow">
        <div class="flex items-center justify-between">
          <div>
            <div class="font-medium">01. ログイン</div>
            <div class="text-sm text-gray-500">メール/パスワード認証画面</div>
          </div>
          <span class="text-gray-400">→</span>
        </div>
      </a>
      <!-- 各画面のリンクを追加 -->
    </div>
    
    <div class="mt-8 p-4 bg-yellow-50 rounded-lg text-sm text-yellow-800">
      ⚠️ これはモックアップです。実際のデータ・API連携は含まれていません。
    </div>
  </div>
</body>
</html>
```

### クライアントへの共有方法
- **ローカル共有**: HTMLファイルをZIPで圧縮してメール添付
- **オンライン共有**: GitHub Pages / Netlify Drop にアップロードしてURLを共有
- **画面録画**: Loomなどで操作説明を録画して共有

## このスキルの典型的な使い方

```
PM: 「design-foundationが終わった。画面確認したい」
 → design-foundationのtokens JSONと画面一覧を貼り付けて起動
 → STEP 1: 対象画面・優先順位確認（止まる）

PM: 「優先順位OK。STEP 2へ」
 → Webリサーチ実施
 → 各画面のHTMLモックアップを生成（止まる）

PM: 「ダッシュボードのレイアウトを変更したい」
 → 修正して再出力

PM: 「STEP 3へ」
 → index.html + フィードバックテンプレートを出力
 → クライアントへの共有方法を案内
```

**設計の原則：**
デザインファウンデーションが確定してからモックアップを作る。色やフォントが決まる前にモックを作ると、「色が違う」「フォントが違う」という無駄な修正が発生する。design-foundationのSTEP 5承認後にこのスキルを起動すること。
