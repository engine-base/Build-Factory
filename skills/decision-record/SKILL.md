## 🧠 全スキル共通：思考品質基準（必ず守ること）

---

### 1. 出力前の必須内部チェック（ユーザーには見せない）

出力を生成する前に、以下を内部で確認する：

- ユーザーの業界・ドメインに固有の法律・規制・制度を参照したか
- 仮説は「売上を上げたい」のような汎用ゴールではなく、そのドメイン・業務フローに固有の仮説になっているか
- 質問は「はい/いいえ」で終わらず、具体例や選択肢を含む設計になっているか
- 曖昧な発言（複数の解釈が可能な表現）に対して複数の解釈を提示したか
- ステークホルダー全員の視点（承認者・反対する人・実際の利用者）を漏らしていないか

---

### 2. 仮説の品質基準

【仮説】ラベルを使う際は以下の形式に従う：

❌ 悪い仮説：「〇〇が良さそう」「〇〇が一般的」（根拠のない選定）

✅ 良い仮説の構造：
- 選定の背景にある制約・要件を具体的に示す
- 複数の代替案を比較検討した上で選定理由を示す
- ポジティブ・ネガティブ両方の影響を明記する

---

### 3. 質問設計の基準

ADRを記録する際は以下を必ず明確にする：
- なぜ今この決定をする必要があるのか（決定を迫られた背景）
- 検討した代替案は何か（最低2案）
- 代替案を採用しなかった理由は何か

---

### 4. ドメインスキャン（出力前に内部実行）

技術選定ADRでは、選定する技術のライセンス・セキュリティリスク・サポート期限も確認する。

---

### 5. 曖昧な発言の複数解釈処理

ユーザーの発言に複数の解釈が可能な場合、解釈を並べて確認する。

---

### 6. ステークホルダーの網羅

ADRの決定者・影響を受けるチーム・将来の保守者の視点を必ず含める。

---

### 7. 質問保留・打ち合わせ優先ルール（全スキル共通）

クライアントから「質問は打ち合わせで」「後で回答します」などの発言があった場合、即座に質問の送信を停止し、現時点の情報を整理して出力する。

---

### 8. 出力フォーマット厳守（最優先ルール）

**スキルモードで動作している場合、出力の冒頭に会話的な前置きを絶対に含めない。**

各STEPの出力は、テンプレートの最初のMarkdown要素（`#`、`##`、`-`、`|` 等）から直接始める。

❌ **禁止（冒頭に付けない）：** 「ありがとうございます」「了解です」「承知しました」「情報を整理します」などの会話的前置き

✅ **正しい出力：** テンプレートの `##` や `|` から直接開始する

**理由：** スキルの出力は `outputMarkdown` としてDBに保存され、プロジェクト管理ドキュメントとして表示される。

---

# 意思決定記録（ADR）スキル

アーキテクチャ・技術選定・プロセスに関する意思決定をADR（Architecture Decision Record）形式で構造化して記録する。タイトル・状況・決定・根拠・影響・代替案・リスクを含む完全なADRをMarkdownとHTMLの両形式で出力する。

**このスキルが解決する問題：**
- 「なぜこの技術を選んだか」が後から分からなくなる
- チームメンバーが変わるたびに同じ議論が繰り返される
- アーキテクチャの変遷が記録されておらず、技術的負債の原因が追えない

---

## ⛔ 絶対ルール

**STEP 1の確認ブロックを出力したら、必ずそこで止まること。**

ユーザーが「STEP 2へ」と指示するまで、絶対に次のSTEPに進んではならない。

---

## テンプレートファイル（assets/）
- `assets/adr-template.md` — ADR単体テンプレート（Markdown版）
- `assets/decision.html` — ADR一覧HTMLドキュメント（タイムライン形式）

STEP 3の最終出力では、Markdown ADRと HTML ADR一覧（decision.html）の両方を生成すること。

---

## ADRステータス定義

| ステータス | 意味 |
|-----------|------|
| 提案中（Proposed） | 議論中・まだ承認されていない |
| 承認済み（Accepted） | チームで合意・採用決定 |
| 廃止（Deprecated） | 過去に採用したが現在は使用していない |
| 置換（Superseded by ADR-XXX） | より新しいADRに置き換えられた |

## ADRカテゴリ

| カテゴリ | 例 |
|---------|---|
| Architecture | システム全体構成・レイヤー設計・通信方式 |
| Technology | 言語・フレームワーク・ライブラリ・DB選定 |
| Process | 開発フロー・ブランチ戦略・テスト方針 |
| Security | 認証方式・暗号化・シークレット管理 |
| Infrastructure | クラウド選定・コンテナ化・CI/CD |

---

## STEP 1: 意思決定の背景確認

**このSTEPでやること：**
記録する意思決定の背景・文脈を整理する。

**確認項目：**
1. **ADRタイトル** — 「〇〇に△△を採用する」形式の短い表題
2. **ADR番号** — 連番（ADR-001, ADR-002...）または任意の識別子
3. **カテゴリ** — Architecture / Technology / Process / Security / Infrastructure
4. **決定者** — この決定を行ったメンバー・役割
5. **決定日** — いつ決定したか（または決定予定日）
6. **ステータス** — 提案中 / 承認済み / 廃止
7. **状況（Context）** — なぜこの決定が必要になったか・どんな制約があるか
8. **決定内容（Decision）** — 何を選んだか・何をすることにしたか

**出力形式：**

```
## ADR基本情報確認

### 識別情報
- ADR番号: ADR-〇〇〇
- タイトル: [「〇〇に△△を採用する」形式]
- カテゴリ: Architecture / Technology / Process / Security / Infrastructure
- ステータス: 提案中 / 承認済み / 廃止
- 決定日: YYYY-MM-DD
- 決定者: [氏名・役割]

### 状況（Context）
（なぜこの決定が必要になったか・どんな問題・制約があるか）

### 決定内容（Decision）
（何を選んだか・何をすることにしたか・一文で）

### 不明点・確認が必要な事項
（代替案・根拠・影響で不明な点）
```

---

📦 **STEP 1 確認**

ADRの基本情報を確認してください。

- タイトルと決定内容の表現は適切ですか？
- 状況（Context）の説明に漏れはありますか？
- 問題なければ「STEP 2へ」とお知らせください

**※ STEP 2には進まない。ユーザーの確認を待つ。**

---

## STEP 2: 根拠・影響・代替案の整理

**このSTEPでやること：**
STEP 1の情報をもとに、ADRの詳細（根拠・影響・代替案・リスク）を整理する。

**整理プロセス：**
1. **根拠（Rationale）** — なぜその決定をしたか。技術的・組織的・コスト的な理由を列挙
2. **ポジティブな影響** — この決定によって何が良くなるか
3. **ネガティブな影響** — この決定によって生じるトレードオフ・制約・コスト
4. **代替案** — 検討した他の選択肢（最低2案）と採用しなかった理由
5. **リスク** — この決定に伴うリスクと軽減策

**出力形式：**

```
## ADR詳細設計

### 根拠（Rationale）
（なぜこの決定をしたか）
- [理由1: 技術的観点]
- [理由2: 組織・チーム観点]
- [理由3: コスト・スケジュール観点]

### 影響（Consequences）

**ポジティブ:**
- [良くなること1]
- [良くなること2]

**ネガティブ（トレードオフ）:**
- [生じる制約・コスト1]
- [生じる制約・コスト2]

### 検討した代替案

| 案 | 概要 | 採用しなかった理由 |
|----|------|----------------|
| 案A（採用） | | — |
| 案B | | |
| 案C | | |

### リスクと軽減策

| リスク | 深刻度 | 軽減策 |
|--------|--------|--------|
| | High/Med/Low | |
```

---

📦 **STEP 2 確認**

ADRの詳細を確認してください。

- 根拠の説明は後から読んでも理解できますか？
- 代替案の比較は公平に記述されていますか？
- 問題なければ「STEP 3へ」とお知らせください（最終出力を生成します）

**※ STEP 3には進まない。ユーザーの確認を待つ。**

---

## STEP 3: 最終出力（Markdown + HTML）

`assets/adr-template.md` と `assets/decision.html` の構造に沿って、確定したADRを2形式で出力する。

### 出力①: ADR（Markdown）
adr-template.md の `{{変数名}}` を全て実際の値で置き換えて出力する。
ファイル名は `docs/adr/ADR-{{ADR_NUMBER}}-{{SLUG}}.md` 形式で保存することを推奨する。

### 出力②: ADR一覧（HTML）
decision.html の `{{変数名}}` を全て実際の値で置き換えて出力する。
タイムライン形式でADRの一覧と詳細を表示するスタンドアロンHTMLとして出力する。

---

## このスキルの典型的な使い方

```
エンジニア: 「Next.jsを採用した理由を記録したい」
 → タイトル・背景を入力して起動
 → STEP 1: 基本情報確認（止まる）

エンジニア: 「STEP 2へ」
 → 根拠・影響・代替案を整理（止まる）

エンジニア: 「代替案にSvelteKitも追加して」
 → 修正して「STEP 3へ」

エンジニア: 「STEP 3へ」
 → 完全なADR（Markdown + HTML）を出力
```
