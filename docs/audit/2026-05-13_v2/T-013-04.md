## Spec literal expansion

ticket text uses generic stub `"as specified by feature F-013"`. F-013 actual
shape (from `docs/functional-breakdown/2026-05-09_v1/features.json`):

> GitHub 連携（PR + HTML diff 注釈）
> happy_path: task 完了 → Claude Code commit + push → PR 自動作成 → HTML レビュー資料添付 → merge で task done.
> **error_paths: "merge conflict → AI 自動解決試行 → 失敗で人間エスカ", "branch protection → 警告 + 別ブランチ"**
> policies: `"force_push": "red line 自動停止"`

T-013-04 covers the bolded error path. Within F-013:

1. **detect**: T-M29-03 `detect_conflict_dry_run` (REUSE, 無改変).
2. **try resolve**: choose a deterministic strategy (`ours` / `theirs` /
   `merge-tree default`) and **re-run dry-run with `-X` favor flag** to see if a
   *clean* tree results. **No actual merge / no push** (red-line: force_push 禁止).
3. **escalate**: if all strategies still produce conflict, return structured
   `escalate_to_human` payload (file list + suggested strategy) which S-033
   PR-review surfaces to the human reviewer.

Therefore T-013-04 = a **thin orchestrator** over T-M29-03:
`detect_conflict_dry_run → try strategies → return AutoResolveResult`.
No new git mutation primitive — purely read-only dry-run combinations.
