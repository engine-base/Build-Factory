ticket text uses generic stub `"as specified by feature F-013"`. F-013 actual
shape (from `docs/functional-breakdown/2026-05-09_v1/features.json`):

3. **escalate**: conflicts が見つかれば即 `status="escalate"` で `next_step="human_review"`.

**T-013-04 (Phase 1) の正直な scope = "detect + escalate"** であり、F-013 spec の
完全実装は Phase 1.5 (T-013-04b) で完成する.

## なぜ Phase 1 で中間段を実装しないか (Phase 境界決定)

| 要因 | Phase 1 での妥当な扱い |
|---|---|
| 真の strategy 試行は `git merge-tree --strategy=ours/theirs/...` (git ≥ 2.40) | git バージョン依存 + dry-run 結果の strategy 差異 evaluation 必要 |
| T-M29-03 `detect_conflict_dry_run` は `-X` flag を受け取らない | REUSE invariant を壊さずに strategy 試行は不可能 |
| LLM-driven 解決は S-5 impact-analysis 連動 | Phase 1.5 の S-5 と整合的にしないと意味が無い |
| Phase 1 SaaS Free tier では LLM コスト管理が未確立 | コスト計測 (T-AI-05) が安定してから着手すべき |

**Phase 1 = detect + escalate のみ** が、F-013 spec を **形式的にも実質的にも整合する**
唯一の Phase 1 答え. これより形式だけ盛れば spot-check で露見した偽装 (v1) になる.

## post-mortem (新方針の限界と修正)

**新 pre-flight audit workflow の限界が見えた事例**:
- audit doc は機械的 1:1 カバレッジは強制できるが、**strategy ラベルが実際に効いているか** は人間 spot-check でしか検出できない
- agent (PR 初版 author) は generic spec を「最小で形式 PASS させる」方向で解釈 → 偽装が generated
- spot-check で発見 → Option B (Phase 縮退) で正直化

**今後の prompt 強化点** (次 wave 以降に反映):
- "strategy / mode / variant 系の引数を取る関数では、各 variant が実際に異なる挙動を発生させる test を 1:1 で書け"
- "形式 PASS させるだけの placeholder code は禁止. 実装できない部分は Phase deferral として明示せよ"
