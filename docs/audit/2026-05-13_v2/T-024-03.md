## Spec literal expansion (REUSE → spec_link 先 で実際に何を要求しているか)

ticket text uses generic stub `"as specified by feature F-024"`. F-024 actual
requirement (from `docs/functional-breakdown/2026-05-09_v1/features.json`):

> グローバル検索（Cmd+K）— Cmd+K で modal → query → Postgres FTS + pgvector +
> pg_trgm → カテゴリ別 → **RLS で見える範囲のみ** → ジャンプ.
> related_entities: tasks, artifacts, audit_logs, knowledge.

T-024-03 `RLS 連動` = global search の各 backend service が **RLS で見える範囲のみ** に
返却を限定する責務を満たす. 具体的には:

1. **per-call workspace/account scope**: `unified_search` / `chat_search.hybrid_search`
   が account_id / workspace_id を受け取って WHERE 句に含める (Phase 1 SQLite
   経路でも同等の filter).
2. **RLS 中央化**: 自前で `auth.uid()` 相当 logic を再実装しない. workspace 判定
   は `bf_can_access_workspace(workspace_id)` (SQL helper, T-001-06) と
   `services.rls_context` (Python helper, T-S0-09b) に delegate.
3. **越権なし**: account_id を指定しない場合でも返却 row は per-source filter で
   絞られ、cross-workspace leakage をしない (= Phase 1 simplification として
   `account_id is None` 経路は admin/service_role 想定で wildcard は OK だが、
   `account_id=N` 指定経路では他 workspace 行が混ざらない).
4. **invalid actor 拒否**: `actor_user_id` が空文字なら 401. account_id<=0 なら 400.
