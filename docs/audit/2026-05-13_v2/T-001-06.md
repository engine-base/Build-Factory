## Spec literal expansion (源泉 cited)

### Ticket title (tickets.json line 1165)

> "RLS 全 23 ユーザデータテーブル enforcement + **custom_permissions 連動**
> (supabase/migrations/20260510000002_rls_full_enforcement.sql / 40 CREATE
> POLICY / bf_can_access_workspace helper / service_role bypass)"

### AC (4 EARS clauses, tickets.json#T-001-06)

1. **UBIQUITOUS**: `20260510000002_rls_full_enforcement.sql` shall enforce
   RLS on **≥ 20 user-data tables** via `ALTER TABLE … ENABLE ROW LEVEL
   SECURITY` and define **≥ 30 RLS policies** via `CREATE POLICY`; include
   a workspace access helper (e.g. `bf_can_access_workspace(ws_id)` SQL
   function joining via `workspace_members`) used by all workspace-scoped
   policies.
2. **EVENT-DRIVEN**: When a user queries a row whose `workspace_id` is not in
   their `workspace_members` set, the RLS policy shall filter the row out
   (zero rows returned) — `USING (bf_can_access_workspace(workspace_id))`;
   migration shall use `DROP POLICY IF EXISTS` before each `CREATE POLICY`
   (idempotent re-application).
3. **STATE-DRIVEN**: While operating as `service_role`, system shall bypass
   RLS via Supabase built-in privilege; migration shall NOT contain blanket
   `FOR ALL TO public` and shall NOT disable RLS (`DISABLE ROW LEVEL
   SECURITY`).
4. **OPTIONAL**: Where the user is `account_owner` via `account_members` or
   `workspace_members` with `role='owner'`, additional policies on
   `accounts` / `workspaces` shall grant cross-workspace read; the
   `bf_can_access_workspace` helper shall be invoked by `USING` / `WITH
   CHECK` clauses (DRY centralization).
5. **UNWANTED**: If RLS is disabled on any user-data table in this
   migration, the spec test shall fail; if a policy is missing `DROP POLICY
   IF EXISTS` counterpart (idempotency violation), the test shall warn;
   migration shall NOT contain any hardcoded JWT (`eyJ...`) /
   `sb_(publishable|secret)_*` / `sk-ant-*`.

### Literal expansion (sub-requirements distilled)

T-001-06 = "**全 user-data テーブルに RLS を enforce し、workspace-scoped
membership を helper 経由で DRY 中央化する S0 critical-path REFACTOR**".

具体的 sub-requirements:

1. **23 user-data テーブル × ENABLE RLS の個別 invariant** (audit prompt
   anti-drift): 21 explicit + alembic_version + 1 legacy 代表 = 23.
2. **service_role bypass per explicit table**: `<table>_service_role`
   policy が `FOR ALL TO postgres, service_role USING (true) WITH CHECK
   (true)` で生成される (Supabase service_role の built-in bypass は
   policy 不要だが、本 migration では明示的 policy で意図を rendered).
3. **idempotency**: 全 explicit `CREATE POLICY` には対応する `DROP POLICY
   IF EXISTS` がある (再 apply で `42710 policy already exists` を避ける).
4. **helper centralization**: `bf_can_access_workspace(workspace_id)` を
   USING 句で >= 5 箇所 invoke (DRY 中央化).
5. **account_owner cross-workspace read**: `bf_is_account_owner(acc_id)`
   helper inline 定義 (`account_members.role='owner'` JOIN), `workspaces`
   の `member_read` policy で OR 条件として混入.
6. **legacy lockdown**: scope 列を持たない 37 legacy single-user table は
   `service_role` 専用 policy のみ付与 (`authenticated` は読み書き不可).
7. **AC-5 negative invariants**: `DISABLE ROW LEVEL SECURITY` / `FOR ALL
   TO public` / `GRANT ALL TO anon|authenticated` / hardcoded JWT/Supabase/
   Anthropic key の **不在**.

  pattern が確立.

- "REFACTOR audit でも、ticket title に動詞 / 名詞句があるなら
  migration source の literal grep を必ず行え (e.g. title が
  `custom_permissions 連動` なら source に `custom_permissions` が
  appear するか grep)"

  「N parametrized assertion」と明示し、prompt の "40-60 tests" 縛り
  との整合を取れ"
