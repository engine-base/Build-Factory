## なぜ "Constitution 注入" を本 module 内で実装しないか (NIH 削減 / ADR-012)

タスク説明には "reviewer persona の system prompt が Constitution 注入される
(M-28 連携)" とあるが、ADR-012 (2026-05-13) 採択により Constitution 注入は
**T-AI-04 → Memory Tool delegate** 経路に統一されている (CLAUDE.md §3 自前実装
8 項目 T-AI-04 注記参照).

| 観点 | Phase 1 での扱い |
|---|---|
| Constitution 注入の責務 | T-AI-04 + Memory Tool (`/memories/constitution/`) 経路 |
| 本 module の責務範囲 | 3-phase persona wrapper の **shape + invariant** のみ |
| 本 module が `constitution_engine` を直接 import するリスク | NIH (Not-Invented-Here) 量産 / Memory Tool delegate と二重実装 / DB read を fallback path に混入させて AC-3 "no DB write" を破る |

→ 本 PR では Constitution 注入 **コードは追加せず**、`constitution_engine` /
`inject_for_session` を本 module から **呼ばない** drift guard test を入れた
(`test_ac3_constitution_inject_not_self_implemented_in_reviewer_persona`).
将来の AI backend hook (T-AI-MEM-01 client-side memory handler) 経由で
Memory Tool が constitution prefix を注入する設計に **準拠** する.

## 3 stage independence の機械検証 (PR #253 lesson)

PR #253 (T-013-04) で発覚した spec drift = "`STRATEGIES = ('default', 'ours',
'theirs')` 全て同一 helper を呼ぶ偽装" を踏まえ、Plan / Gen / Eval 3 stage が
**実際に異なる behavior** を持つことを 1:1 で機械検証:

| 検証軸 | test 関数 | 仕組み |
|---|---|---|
| signature の独立性 | `test_ac2_three_stages_have_distinct_signatures` | `inspect.signature` で input parameter set が互いに disjoint であることを確認 |
| output phase label の独立性 | `test_ac2_three_stages_emit_distinct_phase_label` | `output["phase"]` が `{plan, generate, evaluate}` の 3 distinct value |
| output schema の独立性 | `test_ac2_three_stages_have_distinct_output_schemas` | stage-unique key (dimensions / findings / score) の cross-stage 不在を確認 |
| backend payload key の独立性 | `test_ac2_three_stages_invoke_backend_with_distinct_payload_keys` | hook tracking で 3 payload の `frozenset(keys)` が 3 distinct set |
| 関数 identity の独立性 | `test_drift_guard_plan_and_generate_helpers_are_distinct_functions` | `__code__` object identity check (alias 検出) |
| variant-specific output | `test_drift_guard_each_stage_produces_variant_specific_output` | runtime で stage-unique field の値が生成される |

→ 全 stage が同じ helper を呼んでいたらこれらの test が一斉に fail する.
