## Why a SECOND spec test alongside the existing parametrized invariant?

The existing `test_t_019_01_archive_invariants.py` uses
`@pytest.mark.parametrize("p", ARCHIVED_PATHS)` so a regression on item N
surfaces as the opaque `test_ac1_archived_path_absent[p<N>]`. The audit prompt
explicitly demands **anti-drift**: "each of the 9 archive items MUST have its
own test (not collapsed regex)".

The new module `backend/tests/test_t_019_01_bootstrap_archive_spec.py` therefore:

- Has 9 individually named functions (`test_ac1_item01_onlook_root_dir_absent`,
  `test_ac1_item02_penpot_root_dir_absent`, …, `test_ac1_item09_legacy_adr_0010_filename_absent`).
- A future regression on, e.g., `frontend/src/components/design-canvas/` fires
  as `test_ac1_item04_design_canvas_component_absent` — the failure name itself
  identifies which of the 9 items regressed.
- Complements (does not replace) the parametrized invariant.

## 追加メモ — 「9 archive items」の解釈

tickets.json `existing_files` の 9 行のうち:

- **DELETE 対象** (実 archive): items 1, 2, 3, 4, 5, 6 = 6 ディレクトリ
- **PRESERVE 対象** (guard host): items 7, 8 = 2 ファイル
- **RENAME 対象** (旧 filename 不在): item 9 = 1 ファイル

「9 = 削除数」ではなく「9 = T-019-01 が触る全 path 数」。本 audit はこの
9 item 全てに対する invariant を 1 item / 1 test で機械化する。

将来 someone が item 7 (`lint-mock.sh`) を誤って削除した場合、
`test_ac1_item07_lint_mock_script_present` が即 fail する — preserve invariant
も同 spec module 内で symmetric にカバーしている。
