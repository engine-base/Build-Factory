## なぜ REFACTOR 戦略 = 「既存無改変 + 新規ファイル並置」

| 要因 | この戦略を選んだ理由 |
|---|---|
| 既存 `TaskKanban.tsx` には `groupBy: "feature" \| "status"` の 2 mode が既に実装済み | `groupBy='feature'` mode を使う既存 caller (将来) を壊さない |
| §5.5 の「機能別アコーディオン」は構造が **根本的に違う** (column のネスト = features → 4 status columns) | 既存 flat 6-column board を改修すると 2 つの異なる UI 構造が同じファイルに混在 → maintenance hell |
| CLAUDE.md §5.3 = 「既存実装 (bootstrap) は REUSE/REFACTOR ラベルで活用 (新規追加は本当に必要な場合のみ)」 | "本当に必要" = §5.5 の構造的要請。新規ファイル + 既存無改変 = REFACTOR ラベル下で最小コスト |
| S-028 (list view), S-029 (DAG view) はそれぞれ別 component で実装されている | view 切替は parent 側で component 差替え。アコーディオンも同列に並ぶべき (mode 切替の混入は反 SRP) |

**ergo**: TaskKanbanAccordion.tsx を **新規 file** として追加し、既存 `TaskKanban.tsx`
は touch しない。REFACTOR ラベルは "feature group 内での実装入替" (caller 側で choose)
を意味する範囲で正当化される。
