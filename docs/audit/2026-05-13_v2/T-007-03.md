## post-mortem (anti-drift 強化点)

新 pre-flight audit workflow に対する本タスクからの学び:

1. **REUSE invariant は git diff レベルで assertion すべき**: source-grep だけでは
   "wrapper が substrate を改変していない" の証明にならない (改変後に逆参照を
   削除すれば grep は PASS する). git diff = 0 を機械検証する pattern を
   REUSE audit の標準とする.
2. **wrapper の semantic mapping は AST/body 抽出で assertion する**: literal
   grep は `review_needed` が source に出現すれば PASS するが、それが
   `→ in_progress` mapping を実装しているかは保証しない. 本 audit で
   `_extract_function_body` + regex chain を導入 — 他 wrapper audit にも転用可能.
3. **anti-drift は「実装できないバグ」も検出する**: F1 (edge_type/kind) は
   既存 wrapper コードのバグ. REUSE audit の本来目的は invariant 確認だが、
   spec を 1:1 で literal 検証する過程で自然に露見した. 「fix は別 ticket」
   と documented finding 化することで、 audit の純粋性を保ちつつ問題を
   tracking する.
