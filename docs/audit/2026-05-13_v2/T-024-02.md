### Ticket AC (4 EARS clauses)

1. **UBIQUITOUS**: `backend/services/unified_search.py` (unified_search /
   SOURCE_HANDLERS / VALID_SOURCES / list_valid_sources) と
   `backend/routers/unified_search.py` (POST /api/search/unified +
   GET /api/search/sources) を公開. 4 source (knowledge / tasks /
   employees / screens) を **asyncio.gather で並列** 検索. 既存
   `knowledge_search.py` + `embedding_service.py` は **無改変 (REUSE)**.
2. **EVENT-DRIVEN**: `unified_search(query, sources, account_id, limit_per_source)`
   呼出で dict `{query, sources_used, results[], by_kind, total}` を 2 秒以内に
   返す (各 source 200ms 以内). 各 result item は `{kind, id, label, hint, group}`.
   POST `/api/search/unified` は audit event `search.unified` を emit
   (`query_chars + sources_used + total + by_kind`).
3. **STATE-DRIVEN**: 検索中 **個別 source 失敗で全体は落ちない**
   (`asyncio.gather(return_exceptions=True)` + warning log + skip).
   各 handler は既存 service (embedding_service.search_knowledge / bf_db /
   ai_employee_store) の **thin wrapper** で、新規 DB query は既存 table の
   LIKE のみ. **Read-only** (audit_logs を除き DB write 無し).
4. **UNWANTED**: empty / non-string / > 500 chars な query → ValueError.
   `VALID_SOURCES` 外の source 名 → ValueError. account_id が正の整数で
   なければ 4xx code='search.invalid'. limit_per_source ∉ (0, 50] → ValueError.
   module source 内に hardcoded API key / external URL **無し**.

## post-mortem (本 audit で再確認できたこと)

本 task は **既存 stubbed test (27 件) が形式的には pass している** にも
かかわらず、**実装の 3/4 handler が silent-fail する** という典型的な
"形式 PASS / 実質 fail" のケースだった (PR #253 T-013-04 v1 が同類):

- stub fixture が全 handler を上書きするため、real handler の signature 誤解 /
  import error が test で見えない (Type A gap G1).
- `except Exception: return []` という defensive guard が dataloss を
  silent 化している (AC-3 "isolation" を形式的には満たすが実害は最大).
- drift guard 2 (`measurably distinct kind/group`) は **stub 状態でしか
  意味を持たない** ため、real handler の正常動作は integration test
  (Sprint 4 IT) でしか担保できない.

**Phase 1 honest scope の判断**: 本 audit doc は impl 変更を伴わないため、
T-024-02 自体は "code shape AC × test 通過" で **AUDIT_PASS** だが、
**実機能としては Type C gap 3 件が残る** → フォロー task T-024-02b を
推奨し、parent session に委ねる (cf. T-013-04 → T-013-04b パターン).
