## Spec literal expansion (源泉 cited)

Ticket text uses generic stub `"as specified by feature F-008"`. F-008 actual
requirement (from `docs/functional-breakdown/2026-05-09_v1/features.json`):

> **F-008 プロジェクト・フェーズ管理基盤** (category: moat).
> happy_path: 「案件作成 → Phase 1 default → 新規 Phase 追加 → タスク割当 →
> ゲート条件定義 → ゲート達成で次 Phase auto unlock」.
> error_paths: 「ゲート循環 → DB 制約」「phase 削除 → タスク移動要求」.
> policies: `{max_phases_per_workspace: 10}`.
> related_entities: `phases`, `phase_gates`, `tasks`.

Architecture cross-ref (`docs/architecture/2026-05-09_v1/architecture-v1.md`):
> 「プロジェクト管理 6（**phases** / phase_gates / tasks / task_dependencies /
> acceptance_criteria / constitutions）」

DB schema (`supabase/migrations/20260510000001_bf_project_tables.sql`):
> `CREATE TABLE bf_phases (id BIGSERIAL PRIMARY KEY, project_id BIGINT NOT NULL
> REFERENCES bf_projects(id) ON DELETE CASCADE, phase_no INTEGER NOT NULL CHECK
> (phase_no BETWEEN 1 AND 10), name TEXT NOT NULL, status TEXT NOT NULL DEFAULT
> 'pending' CHECK (status IN ('pending','in_progress','completed','blocked','skipped')),
> artifacts_dir TEXT, started_at TIMESTAMPTZ, completed_at TIMESTAMPTZ, notes TEXT,
> CONSTRAINT uq_bf_phase UNIQUE (project_id, phase_no))`

Therefore T-008-01 `phases CRUD` = F-008 happy-path の "Phase 追加 / 状態遷移
(default → 進行中 → 完了) / soft-delete" を実現する **backend service + router** が
DDL (T-001-04) の 5 制約 (phase_no 1..10 / status enum / unique (project_id,
phase_no) / FK project_id / name NOT NULL) を満たし、ticket AC 4 種 (UBIQUITOUS /
EVENT-DRIVEN / STATE-DRIVEN / UNWANTED) を機械検証可能にすることである. 具体的な
sub-requirements:

1. **CRUD 7 endpoint** : list / get / create / update / start (lifecycle) /
   complete (lifecycle) / soft-delete (status='skipped') が REST router で公開.
2. **DDL 整合性** : service layer の validation (`_validate_phase_no` /
   `_validate_status` / 空 name 拒否 / UNIQUE 違反検知) が migration の CHECK /
   CONSTRAINT 制約と 1:1 で対応.
3. **error envelope** : 全 4xx error は `{detail: {code, message}}` 形式を返却し、
   code は `phase_not_found` / `phase_duplicate` / `invalid_name` /
   `invalid_status` / `invalid_phase_no` の 5 種 (router 内 doc-string で公開).
4. **idempotent lifecycle** : `start_phase` が既に in_progress の row に対して
   second-call で no-op (state mutation を起こさない). 同様に soft-delete
   (`status='skipped'`) は physical DELETE しない (F-008 error_path 「phase 削除
   → タスク移動要求」を満たすために dependent task の orphan を防ぐ).
5. **status mutation gate** : invalid input (`InvalidPhaseInput`) は INSERT /
   UPDATE の execute 前に raise (4xx + no state mutation の UNWANTED AC).
6. **REFACTOR invariant** : T-001-04 で確立した bf_phases DDL は改変しない.
   service / router の公開 symbol を保つ (T-008-02 FE / T-008-03 gate /
   T-009-01 dependencies が同 schema を参照).

## Phase 1 vs Phase 2 境界決定

- **Phase 1 (現在)**: CRUD service + router は SQLite (`db.async_db`) 経路で動作.
  RLS は test 内では bypass. `started_at` / `completed_at` は SQLite
  `datetime('now','localtime')` (TIMESTAMPTZ は Phase 2 Postgres で復活).
- **Phase 2 (deferred)**:
  - audit_logs trigger による phases CRUD の自動 INSERT (G3 deferred).
  - F-008 `phase_gates` 配下の循環検出 DB 制約 (T-008-03 で部分実装).
  - `max_phases_per_workspace` の運用上の上限 = `PHASE_NO_MAX=10` で代替 (workspace
    全体ではなく project 単位; F-008 policy の文言上 workspace と project の用語が
    混在しているが、DDL は project_id FK で project スコープ).
