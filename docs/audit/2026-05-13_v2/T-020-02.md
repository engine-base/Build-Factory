### Ticket AC (6 EARS clauses)

1. **UBIQUITOUS** : `memory_service.py` shall expose a unified async API combining Tier 1/2/3 with min 6 public symbols (`emit_event, persist_compaction, write_fact, merge_for_session, mirror_to_obsidian, fact_fingerprint`).
2. **EVENT-DRIVEN** : When SDK auto compaction triggers at 95%, `persist_compaction(session_id, summary)` shall (1) write 9-section summary to `chat_messages.compressed_summary` JSONB and (2) emit `memory_compacted` event to `audit_logs` within 2s. Runner shall NOT generate summary itself.
3. **EVENT-DRIVEN** : When `merge_for_session(session_id, prior_session_id, user_id, ...)` is invoked, merge SDK session_resume + Memory API recall + Mem0 top-k + Constitution deterministically.
4. **STATE-DRIVEN** : While Memory API available, `write_fact` shall write Memory API as primary + Mem0 copy. `SECTION_KEYS` shall be defined exactly once in `mid_term_layer.py` with 9 entries; NOT redefined in `memory_service.py` (G15 cross-module invariant).
5. **OPTIONAL** : Where user opts in to Obsidian sync, mirror Tier 3 facts to Vault as Markdown with deterministic `fact_fingerprint(text)` sha256-prefix file names.
6. **UNWANTED** : If Memory API write fails, `write_fact` shall fall back to Mem0 only + emit `memory_degraded` event with reason. NOT silently drop facts.

## なぜ pre-flight audit を行うか (PR #253 lesson — anti-drift critical)

> "strategy / mode / variant 系の引数を取る関数では、各 variant が実際に異なる挙動を発生させる test を 1:1 で書け"
> "形式 PASS させるだけの placeholder code は禁止"

T-020-02 は **3 tier** という variant 概念を持つ. PR #253 (T-013-04) の偽装事例 (`STRATEGIES = ("default", "ours", "theirs")` が全 strategy で同一 helper を呼ぶ) と同型の罠が成立しうる:

> **「3 tier が同じ helper を呼ぶだけ偽装」の罠**
> 例: short / mid / long が全て `chat_thread_store.list_messages()` を内部で呼ぶだけで, dict の key 名だけ違うが書込先・読込先・schema が同一になっていないか?

本 audit はこの罠を機械検知するため, 各 tier の以下を **個別 verify** する:
1. **output schema** が cross-tier で **完全に disjoint** な signature key を持つ
2. **write target** が cross-tier で disjoint (Tier 1 = read-only / Tier 2 = chat_thread_store + persist_compaction / Tier 3 = mem0 + filesystem)
3. **read source** も disjoint (Tier 1/2 = in-memory store / Tier 3 = mem0 vector + filesystem glob)
4. 各 tier は独自 Error class を持ち, 互いに継承していない
5. ADR-010 禁則 (LangGraph / LangChain / LiteLLM) が import 文に出現しない

## 3 tier 不変条件 (1 表で対比)

| 側面 | Tier 1 (Short) | Tier 2 (Mid) | Tier 3 (Long) |
|---|---|---|---|
| **module** | `short_term_layer.py` | `mid_term_layer.py` | `long_term_layer.py` |
| **write target** | NONE (read-only) | `chat_thread_store.add_message` (route A) + best-effort `memory_service.persist_compaction` (route B / sqlite) | Mem0 `add_conversation` + filesystem `{root}/{user_id}/{src}-{ts}-{suf}.md` |
| **read source** | `chat_thread_store.list_messages` (in-memory) | 同左 + `_classify_source` で 2 route 統合 | Mem0 `search_relevant_memories` + Obsidian `glob("*.md")` token-overlap |
| **output dict signature key** | `messages`, `n`, `exclude_summaries`, `role_filter`, `count` | `summary` (9-section), `prefer_source`, `found`, `source` (route name) | `scopes`, `results`, `tags`, `user_id`, `status`, `source` (enum) |
| **N / size constants** | `DEFAULT_FIFO_N=20`, `MIN=1`, `MAX=200` | `MAX_HISTORY_LIMIT=200`, `MAX_FETCH_MESSAGES=10_000` | `MAX_TOP_K=100`, `MAX_CONTENT_CHARS=50_000`, `MAX_TAGS=20` |
| **Schema fields** | raw `{id, role, content, created_at, has_compressed_summary}` | 9 SECTION_KEYS (context/goals/decisions/open_questions/actions/blockers/facts/preferences/next_steps) | source enum (conversation/fact/decision/knowledge/constitution) + tags + YAML frontmatter |
| **Error class** | `ShortTermLayerError` | `MidTermLayerError` | `LongTermLayerError` |
| **SECTION_KEYS** | ❌ 参照しない (G15) | ✅ 唯一定義 | ❌ 参照しない |
| **直接書込先 module** | なし | `chat_thread_store` のみ | `services.long_term_memory` + filesystem |

> **注**: `source` キー名は Tier 2 と Tier 3 の両方に出現するが, semantics は完全に異なる (Tier 2 = route name `"compressed_summary"|"system_summary"`, Tier 3 = source enum `"conversation"|"fact"|...`). 同名キーでも値ドメインが disjoint なため drift ではない. `test_cross_tier_output_dicts_have_disjoint_signature_keys` の signature set からは除外する.

## post-mortem (本 audit の学び)

**何が良かったか**:
- PR #253 の偽装事例を spec 化し, `test_cross_tier_output_dicts_have_disjoint_signature_keys` + write-target grep + Error class identity check の 3 段で機械検知できた
- "同名異義" の `source` key を「signature set から除外 + doc に明記」で誤検知ゼロにできた
- REFACTOR の signature lock を `inspect.signature` で 6 件固定 (default 値含めて) — 将来の "ちょっとだけ変更" を impossible に

**今後の prompt 強化点**:
- 「3 tier / 3 phase / 3 mode のような variant を持つタスクでは, 各 variant の output schema / write target / read source を cross-variant disjoint で機械検証する」 → 本 audit doc の Anti-drift cross-tier guard セクションを標準テンプレ化候補
- "同名異義キー" は audit doc 内で明示する (signature set から除外する根拠とともに) — 誤検知ゼロのための必須手順
