## post-mortem (audit 設計レッスン)

**Wave 5 anti-drift rule の効用**:
- 17 個別 test 関数を機械的に列挙すると、PR diff レビューで「あ、〇〇テーブルの test が消えてる」が一目で分かる。
- 既存 parametrize/loop ベースの test では「17 中 1 つ削った」regression が "1 case failed" としか出ず、視覚的に弱い。
- 個別 test は冗長に見えるが、**drift 検出の粒度** = **test 関数の粒度** という invariant が成立する。

**enum exact-set 検査の効用**:
- 「期待値が含まれる」だけだと additive drift (値が増える) を見逃す。
- `set(...) == expected` で additive / subtractive 両方向の drift をブロック。
- 5 つの enum (template_kind / backup status / backup_kind / triggered_by / body_format) すべてに適用。

**今後の test 追加**:
- T-001-05 は production 適用 (`supabase db push`) 前なので、live DB 制約検証 (実 INSERT で NOT NULL / FK reject) は Sprint 0 終盤の T-001-11 (`test_t_001_11_db_integration.py`) で実施予定。本 audit は静的 SQL 検証で止め、live は別 ticket に切り出してある。
