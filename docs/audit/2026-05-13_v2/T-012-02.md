## なぜ各 pattern を個別 rule に分けるか (drift guard 設計理由)

T-013-04 v1 で発覚した spec drift = 「3 strategy 並ぶが同一 helper を呼ぶ
偽装」と同じ罠が、ここでも起こりやすい:

```python
# ❌ ダメな実装 (drift)
SQL_REDLINE_PATTERN = r"\b(DROP|TRUNCATE|DELETE)\b"  # 1 個の generic regex
def detect(text): return bool(re.search(SQL_REDLINE_PATTERN, text))
```

これだと:
- どの red-line が hit したか分からない → audit log で原因特定不可
- `severity` を rule 単位で変えられない (DROP=block / log=warn 等)
- false-positive (`DROP TABLE IF EXISTS`) を rule 単位で除外不可
- F-012 default_categories 5 種が "category" として上がってこない

本 PR では:
```python
# ✅ OK な実装
RULE_SQL_DROP = RedLineRule(rule_key="no_drop_table", category="db_destructive",
                            pattern=r"\bDROP\s+(?:TABLE|...)(?!\s+IF\s+EXISTS)\b",
                            severity="block", ...)
RULE_SQL_TRUNCATE = RedLineRule(rule_key="no_truncate", ...)
RULE_SQL_DELETE_STAR = RedLineRule(rule_key="no_delete_star",
                                   pattern=r"\bDELETE\s+FROM\b ... (?!WHERE)", ...)
# ... 計 11 個別 rule
```

各 rule:
- `rule_key` unique
- `category` ∈ F-012 default_categories
- `severity` ∈ {block, warn, log}
- 自前 regex で WORD-BOUNDARY + negative lookahead により false-positive guard

drift guard test (`test_ac1_individual_rule_instances_exposed` /
`test_ac1_rule_keys_unique` / `test_ac1_db_destructive_has_three_individual_rules`)
が「個別 rule であること」を pattern matching ではなく **module symbol
存在確認** で守る.
