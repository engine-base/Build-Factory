## ノート

### 実装方針 (impl 時の判断)

1. **既存 router/service の活用**: `backend/routers/accounts.py` に 3 endpoint (transfer-owner / invitations / member removal) を追加。
   workspace 側の F-004 invitation 系は `routers/workspaces.py::invitations_router` で既に充実しており、
   そちらは温存。新規 public `GET /api/invitations/{token}` のみ `backend/routers/invitations.py` を新設。
2. **account_invitations table**: workspace_invitations と並列の新テーブルを alembic `i6d7e8f9a1b2_account_invitations.py` で追加。
   account_service には test/dev で alembic 未適用でも動くよう `_ensure_account_invitations_table()` lazy DDL を実装。
3. **rate-limit (AC-F2 / AC-F10)**: redis 未使用環境のため `backend/services/invitation_service.py` に
   in-memory sliding-window バケットを実装 (20 req / 1 hour / account)。テスト用に `configure_rate_limit()` /
   `reset_rate_limit()` を export し、CI でデフォルトに復元される。本番 Redis 化は次 Wave で差し替え可。
4. **auth (AC-F5 / AC-F8 / AC-F12)**: 既存 `services.auth_middleware.require_user` を `Depends` 経由で接続。
   `BUILD_FACTORY_DEV_BYPASS_AUTH=0` で 401 が動作することを smoke で確認。
5. **public lookup (AC-F13 / AC-F3)**: `invitation_service.public_lookup()` で account-level → workspace-level
   の順に解決。is_expired / status=accepted は 409 で返す (status=pending のみ 200)。

### Endpoint summary
- POST `/api/accounts/{id}/transfer-owner` (201, owner_user_id atomic 移譲, audit emit)
- POST `/api/accounts/{id}/invitations` (201, rate-limit 20/h/account, audit emit)
- DELETE `/api/accounts/{id}/members/{user_id}` (200, owner 削除は 409)
- GET `/api/invitations/{token}` (200 public, account → workspace 両対応, 409 if expired / accepted)
