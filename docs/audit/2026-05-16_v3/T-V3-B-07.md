## ノート

### 実装方針
- F-005 で要求される 4 endpoint:
  - `POST /api/workspaces/{id}/hearing/save` → backend/routers/hearing.py に追加 (既存 hearing endpoint と共存)
  - `GET  /api/workspaces/{id}/specs` → backend/routers/specs.py (新規)
  - `GET  /api/workspaces/{id}/specs/{spec_id}/comments` → backend/routers/specs.py
  - `POST /api/workspaces/{id}/specs/{spec_id}/comments` → backend/routers/specs.py
- 内部 helper として `POST /api/workspaces/{id}/specs` (spec 作成) も提供 (test fixture / hearing→spec パイプライン用)
- 永続層: hearings / specs / comments テーブルは v3 phase1 時点で migration 化されていない. 本 task では `backend/services/specs_store.py` を thread-safe in-memory store として提供し、F-005 contract と 3-tier AC の挙動 (slot_state monotonic version / paused→resume / 10000 char limit / workspace isolation) を満たす. Postgres 実装 + RLS は Phase 1.5 で置換予定 (drift note T-V3-DRIFT-F-005-* の解消は段階的に行う).
- 認証: `services.auth_middleware.require_user` を Depends で組み込み、401 を担保. 既存テストとの整合を保つため `BUILD_FACTORY_DEV_BYPASS_AUTH=1` 時は DEV_USER で通過 (legacy behavior 不変). 401 ケースは test 内で監視済み.
- E-009 (Hearing) / E-010 (Spec) / E-030 (Comment) は in-memory dataclass で表現. ChatThread (E-027) は本 task では touched せず chat_thread_store / hearing_service の既存実装と共存.
