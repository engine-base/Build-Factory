## 補足項目 (タスク固有)

- **work_package_boundary 遵守**: editable は (1) `supabase/migrations/20260516190000_critical_new_entities.sql` (new), (2) `backend/app/models/{skill_execution,phase_gate,user_knowledge_namespace}.py` (new), (3) `backend/tests/integration/test_critical_new_entities.py` (new), (4) `docs/functional-breakdown/2026-05-16_v3/entities.json` (modify). forbidden の `frontend/` は一切触らない. 共有書込み回避: `entities.json` は `shared_no_concurrent_edit` にあるが本 task で他 task と並列衝突する箇所は無い (E-009/13/10 のみ更新).
- **layout 逸脱の記録**: ticket の `files_changed` は `backend/app/models/{...}.py` を SQLAlchemy model 指定. しかし Build-Factory の backend は raw SQL via psycopg (`backend/db/async_db.py`) を採用しており SQLAlchemy `DeclarativeBase` は存在しない. 本 task では SQLAlchemy を新規導入する spec-change 規模の refactor を避けるため、 同 path に **Pydantic `BaseModel` (Create + Full row) を配置** する形で対応した. これにより work_package_boundary を満たしつつ project-wide pattern (Pydantic schemas in `backend/schemas/`) と整合する typed contract を future API endpoint (`/api/skills/{id}/test` 等) に提供する. `backend/app/models/__init__.py:1-49` にこの判断を記録. (CLAUDE.md §11 「軽微変更」境界内: spec の意図 (3 entity に正式 model 定義) は満たし、 重大変更 (主要パッケージ追加 = SQLAlchemy 導入) は回避)
- **layout / FK 設計上の判断**: PhaseGate spec の tenant_isolation は workspace_scoped (workspace_id) と記載されているが、 v1 fields[] は phase_id のみ. 本 task では (a) phase_id (FK to bf_phases, 自然な親) と (b) workspace_id (FK to workspaces, 非正規化) の両方を保持し、 RLS は workspace_id 直接参照で評価する設計を採用. これにより access_policies_required の `phase_gates_workspace_member_select` を bf_can_access_workspace(workspace_id) でシンプルに評価可 (D-07 の bf_features / bf_mocks のような 2-hop join を回避). migration header (`:97-109`) + audit MD に記録.
- **scope enum 拡張の記録**: E-010 UserKnowledgeNamespace の spec は `scope enum(private/shared)` だが、 ticket AC-F4 は `(private/account/workspace)` を要求. 本 task は ticket AC を一次ソースとし、 (private/account/workspace) で CHECK 制約を実装. これは Mem0/Obsidian/Constitution namespace isolation のため account レベルと workspace レベルの分離が必要との上流意図に整合 (T-AI-MEM-04 provider-adapter precedence と一致). migration header (`:147-152`) に rationale を記録.
- **entities.json 反映**: E-009/E-013/E-010 の status を `decided` → `formalized_in_migration` に遷移. fields[] / access_control_policies[] / indexes[] / constraints / purpose / policy_count_in_migration=2 を埋め直し. legacy_drift_notes.impl_table を `(missing)` → `<table_name>` に修正、 resolved_by_task=`T-V3-D-12` / source_migration=`20260516190000_critical_new_entities.sql` を記録. これにより drift summary §2「Critical drift 3 件」が closed.
- **idempotency**: 全 CREATE TABLE は `IF NOT EXISTS`. 全 CREATE POLICY は `DROP POLICY IF EXISTS` 前置. schema_versions INSERT は `ON CONFLICT (version) DO NOTHING`. 全 CREATE INDEX は `IF NOT EXISTS`. UNIQUE constraint は CONSTRAINT 名付きで CREATE TABLE 内に宣言 (CREATE TABLE IF NOT EXISTS で table 自体が既存ならスキップされ、 重複 apply 時にも安全).
- **security baseline**: `DISABLE ROW LEVEL SECURITY` 一切使用なし (`test_no_disable_row_level_security` で静的検証) / `FOR ALL TO public` 一切使用なし (`test_no_for_all_to_public`) / 全 3 table で service_role 経路のみ書込み可、 authenticated は workspace_member_select (skill_executions / phase_gates) or owner_only (user_knowledge_namespaces) で読取制限.
