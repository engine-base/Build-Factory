## ノート

- 既存 `frontend/src/api/search.ts` (T-V3-C-11 で確立済) を REUSE。`SearchApiError.toUserMessage()` のエンドポイント suffix 付き friendly message 規約をそのまま AC-F1 対応に利用 (NIH 抑制)。
- 検索 API は `?category=` を **1 値** しか受け付けないため、UI 側の multi-select facet checkboxes は次のロジックで backend に橋渡しする: (a) ちょうど 1 つ選択時はその category を `?category=` で送る、(b) 0/2+ 選択時は param を省いて広めに取得しクライアント側で post-filter する。これは backend 仕様 (openapi.yaml#/api/search) と mock の multi-select UI の両方を破壊しない妥協点。
- 200ms debounce + `staleTime: 30s` で 60 req/min rate-limit に余裕を持たせた (S-011 と同じ tuning)。
- ranking 順序は backend `hits[]` の配列順をそのまま render することで保証 (sort 介在なし)。AC-F2 = AC-F3 の保証。
- 日付 facet (過去 7d / 30d) は backend 未対応のため UI 状態としてのみ保持 (現時点で server に渡さず、TODO は backend ticket 化されるまでローカル表示のみ)。lint-mock の DRIFT 規則は spec/mock 上の項目を render していれば OK。
- task spec の `files_changed` 一部は `frontend/src/app/search/results/page.tsx` と書かれているが、`work_package_boundary.editable` が `frontend/src/app/(app)/search/results/page.tsx` (App Router の `(app)` route group) を許可しているのでこちらに配置。`(app)` 配下に既に `search/page.tsx` が存在しレイアウト統合が自然。
