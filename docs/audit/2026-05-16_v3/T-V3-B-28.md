## ノート
- audit MD の `backend/app/...` パスはテンプレ由来. 実 backend は `backend/routers/...`
  のフラット構造 (`backend/app/` ディレクトリは存在しない). 実装は実構造に従った.
- E-042 Constitution / E-031 RedLine と audit MD は記載するが, 実際の
  entities.json の ID は **E-017 Constitution / E-018 RedLine** (E-042 は ChatMessage,
  E-031 は SlackWebhook). 実装は table 名 (bf_constitutions / bf_constitution_revisions
  / red_lines) を基準にしたため影響なし.
- bf_constitutions は project-scoped (project_id) だが API は workspace-scoped.
  既存 constitution_engine.py の慣例に倣い workspace_id を直接 row scope key として
  運用. 本 PR の services/constitution_service.py は新規 sqlite テスト環境向けに
  workspace_id 直結スキーマ (`_ensure_tables` で bootstrap) を採用しつつ, 既存
  supabase migration の bf_constitutions schema とも互換 (column 共通 + idempotent
  `CREATE TABLE IF NOT EXISTS`).
- principles JSONB ⇔ content_md 単一文字列の翻訳は `_principles_to_content_md`
  /  `_content_md_to_principles` で実装. legacy section_* キーからの fallback も
  サポート (`test_principles_to_content_md_section_fallback`).
- AC-EVENT (cache flush): approve 時に constitution_engine.invalidate_cache() を
  best-effort 呼出 (失敗は warn-log).
- auth: 本 repo は Bearer JWT 未配備. workspaces.py の "soft auth" pattern に
  倣い `user_id` (GET) / `actor_user_id` (POST) を token surrogate として扱う.
  本番 deploy 時に Supabase Auth middleware を上乗せする前提.
