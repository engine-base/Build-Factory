## ノート
- 実装パスの差異: ticket は `backend/app/routers/...` 想定だが、本リポジトリは `backend/routers/...` 構造 (app/ サブパッケージなし). 既存 audit_trigger.py / cost_dashboard.py と整合させて `backend/{routers,services,schemas}/audit_logs.py` に配置.
- auth dependency 設計: 既存 router は auth_middleware を使っていない (DEV_BYPASS=1 既定). 本 router で `_require_user_dep` を **call 時 lazy import** にし、SUPABASE_* env が未設定でも router module 自体は import 可能 (T-001-01 invariant 遵守). main.py 全体の起動には影響しない.
- 90-day window 制約: features.json は CSV/JSON export のみに >90 day 制約を載せているが、 list endpoint にも一律で同じ validation を適用 (AC-F4 = "request body failing validation"). validate_date_range は from / to を dual-bound にして単片指定を 422 filter_too_broad で弾く.
- CSV 出力: features.json は `{csv_body: string}` を要求するが、 UX 上は `text/csv` ストリーム & `Content-Disposition: attachment` がデフォルト. `?as=json` または `Accept: application/json` で JSON wrapper を返す negotiation を実装.
- DB layer: cost_dashboard pattern の graceful empty fallback (`_safe_fetch_rows`) を再利用. table 不存在 / 接続失敗で `[]` を返し 200 で応答する設計 (Phase 1 で実 DB 未接続環境を許容).
- 既存 audit_trigger は別 router (`/api/audit-triggers`) として継続稼働. 本 router (`/api/audit-logs`) と path prefix が衝突しないことを `test_router_registered_in_main` で確認.
- pyright `_CursorWrapper.description` 警告は cost_dashboard.py と同じ既知問題. `getattr` 経由で回避.
