- 実装パスの差異: ticket は `backend/app/routers/...` 想定だが、本リポジトリは `backend/routers/...` 構造 (app/ サブパッケージなし). 既存 audit_logs.py / cost_dashboard.py と整合させて `backend/{routers,services,schemas}/notifications.py` に配置.
- auth dependency: 既存 audit_logs router と同じく `_require_user_dep` を **call 時 lazy import** にし、SUPABASE_* env が未設定でも router module 自体は import 可能 (T-001-01 invariant 遵守).
- recipient_user_id 抽出: DEV_USER の `sub` claim (UUID) を notifications.recipient_user_id (TEXT) として使う. DB の RLS policy (notifications_recipient: `recipient_user_id = auth.uid()::text`) と整合.
- read-all category filter: features.json は payload に `category` (nullable) を持つので、event_type prefix (e.g. "pr.*" → "pr.%") として LIKE で絞り込む. None / "" → 全 unread 対象 (AC-F2).
- 単一 read 冪等性: 既読 row への重複 POST でも 200 + read_at を返す (idempotent). row 不存在 / 他人の row のみ 404.
- DB layer: audit_logs service と同じ `_safe_fetch_rows` / `_safe_execute` graceful empty fallback (table 不存在 / 接続失敗で空 list / 0). Phase 1 で実 DB 未接続環境を許容.
