- Actual project layout uses `backend/{routers,services,schemas,tests}/`, not `backend/app/...` as the ticket spec implies. Followed real layout; all 80+ existing routers use this convention. Module imports (`from schemas.auth import ...`) are consistent with the rest of the codebase (e.g. `from services.oauth_providers import ...` in `backend/routers/oauth.py`).
- Test run: **24/24 PASS** in 14.95s. Coverage: routers/auth.py 90%, services/auth.py 97%, schemas/auth.py 100%, TOTAL 95%. Pyright strict on touched files: 0 errors.
- 13 AC-mapped tests (1 per Tier 2 functional AC) + 4 additional regression tests for bucket isolation, duplicate-email 409, and parametrized validation cases. Parametrized cases expand to 24 effective test cases.
- Rate limiter uses sliding-window with `asyncio.Lock`. Per-`(scope, ip)` buckets isolate signup / login / password-reset and per-client. `Retry-After` header set on 429.
- AC-F2 (no user enumeration): both "unknown email" and "wrong password" raise the same `InvalidCredentialsError`, producing identical 401 body/code/message. Test `test_ac_f2_*` asserts byte-equal `detail.code` and `detail.message` across both paths.
- AC-F3 (no account enumeration on password-reset): always returns 200 with the same body regardless of account existence; service returns a bool only for internal audit (discarded by router).
- The Pydantic v2 `str` field with `pattern=_EMAIL_PATTERN` (lightweight RFC 5322) enforces email shape without forcing the optional `email-validator` (`pydantic[email]`) dependency at import time. On validation failure FastAPI emits 422 with `detail[*].loc=["body","email"]` field-level map (verified by `test_ac_f6/f9/f12_*`). MX / DNS-level validation is delegated to Supabase Auth (GoTrue) layer in production.
- Implementation is REUSE-aware: depends only on existing `services/supabase_client.py` patterns (no new framework). Production replacement (Supabase Auth GoTrue) is deferred to T-V3-B-02+; this task delivers spec-conformant interface + business logic + tests.
