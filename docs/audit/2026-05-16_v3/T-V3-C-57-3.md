## Tier 2: Functional

- [x] AC-F1: EVENT-DRIVEN: When a user changes any filter input, the system shall debounce 250ms and then re-call GET /api/workspaces/{id}/tasks?group_by=feature with the corresponding query parameters; the URL search params shall mirror the active filter state. → impl: `frontend/src/app/(app)/task/kanban/use-kanban-filter.ts` `useKanbanFilter` — `useEffect` timer with `KANBAN_FILTER_DEBOUNCE_MS = 250` ms commits `state` into `debouncedState`; a second `useEffect` calls `window.history.replaceState(null, "", "/path?<filterStateToSearchParams(debouncedState)>")` so the URL search params always mirror the committed state. The data fetcher (T-V3-C-57-1) watches `debouncedState` to trigger refetch.
- [x] AC-F2: STATE-DRIVEN: While a filter narrows results to zero, the system shall display an empty state per accordion section with a 'Reset filters' CTA. → impl: `frontend/src/app/(app)/task/kanban/filter.tsx` `KanbanFilterEmptyState` (`data-testid="kanban-filter-empty"`, `role="status"`, `aria-live="polite"`, `kanban-filter-reset` CTA) + page wiring at `frontend/src/app/(app)/task/kanban/page.tsx` (`visibleFeatures.length === 0 ? <KanbanFilterEmptyState onReset={filter.reset} /> : …`).
- [x] AC-F3: UNWANTED: If text search input exceeds 200 characters, the system shall truncate to 200 and not call the API beyond that length. → impl: `truncateQuery` + `KANBAN_SEARCH_MAX_LEN = 200` in `frontend/src/app/(app)/task/kanban/filter.tsx`; applied in `handleQueryChange` and as the `<input maxLength={KANBAN_SEARCH_MAX_LEN}>` attribute. The hook `setQuery` re-applies `truncateQuery` so `debouncedState.query` is structurally incapable of exceeding the cap → the downstream API call (T-V3-C-57-1 fetcher) never sees a longer payload.
