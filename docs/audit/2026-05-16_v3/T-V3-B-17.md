## ノート
- **layout discrepancy**: audit MD / tickets.json の `files_changed` は `backend/app/routers/...` を指すが、実 repo layout は `backend/routers/...`。後者は v1/v2 全期間で 100 件超の router が同一構造で配置されており、layout 改名は破壊的変更 (work_package_boundary の `shared_no_concurrent_edit: backend/app/main.py` も実体は `backend/main.py`)。本 task では実 layout に合わせて実装し、boundary 趣旨 (red_lines.py / violations.py / red_lines service / red_lines test) は維持。
- **REUSE invariant**: 既存 `backend/services/red_line_detector.py` (T-012-02) の DEFAULT_RULES / detect_patterns を service 層から呼び、新規 regex を増やさず drift guard を遵守。
- **store**: Phase 1 baseline は in-process (threading.RLock + dict) で persistence。Wave 2 以降の workspace-aware Postgres backend (E-031 / E-032 RLS 既存) への置換時、`services/red_lines.py` の `_RED_LINES` / `_VIOLATIONS` dict を SQLAlchemy async session に切替する設計 (reset_store() 互換)。
- **auth**: 3-header model (`Authorization: Bearer` + `X-User-Id` + `X-User-Role`)。`X-User-Role=workspace_admin` が POST red-lines / approve / reject / GET violations の必須条件。Phase 1 SSO / Supabase JWT 統合時に `_validate_auth` を JWT decoder に差し替え予定。
- **endpoint count**: tickets.json は endpoint_count=3 だが F-012 全体は openapi.yaml で 6 endpoint (GET/POST/POST red-lines + POST test + GET violations + POST approve/reject)。本 task では F-012 backend を完結させるため 6 endpoint 全て実装 (approve/reject は audit MD AC-F4/F5 の必須実装)。violations.py の reject + GET workspace violations / GET violation by id は Wave 1 で同時提供することで、後続 UI task (T-V3-C-* violations_inbox) のスタブ待ちを発生させない。
- **endpoint paths**: openapi.yaml は `backend/routers/workspaces.py::*` を implementation-path として宣言するが、F-012 を `workspaces.py` (834 lines / 12+ endpoint 既存) に追加すると domain boundary が崩れるため、F-012 専用 router `backend/routers/red_lines.py` + `backend/routers/violations.py` に分離。openapi.yaml の `x-bf-implementation-path` は本 PR 範囲外 (`docs/api-design/` は readonly) で、次 Wave の api-design refresh で同期予定。
