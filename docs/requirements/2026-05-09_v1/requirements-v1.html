プロジェクト概要
目的

Build-Factory（開発工場 OS） = 企画から納品までの全工程を 1 つの Web アプリで完璧に進められる SaaS 型開発 OS。

核心課題：AI 開発が流行する一方、管理 / 品質の不足で「漏れ」「使えないものになる」事故が頻発している。Build-Factory はこの管理・品質の壁を AI チームと管理盤の一体化で破壊する。

本質的なゴール：1 人で 10 案件並走（1 案件 5 並列タスク）しても破綻しない開発体制を作り、SaaS 商用化する。

現状の課題と影響

M-1Supabase 基盤構築 + テストデータ seedPhase 1
概要
Supabase Postgres + Auth + RLS を 0 から新規構築。既存ユーザ移行なし。テストデータ seed を投入。
入力
  • Supabase プロジェクト credentials
  • 新規スキーマ DDL
  • seed JSON（test_account / 6 ロール test_user / 5 sample task / 3 artifact / 2 AI 社員）
処理
1. Supabase 新規作成 → DDL 適用 → RLS 全テーブル enable
2. seed スクリプト実行（BF_ENV=development のみ）
3. Storage に sample HTML upload + Realtime 統合
エラー
RLS 漏れ → 自動テスト追加 / seed ID 衝突 → UUID + 冪等性
制約
標準 SQL 維持 / production では seed 走らない
M-3AI 社員ハイブリッド統合（BMAD + Agent Teams + 既存スキル）Phase 1
概要
BMAD 12 ペルソナ + Anthropic Agent Teams + 既存 96 スキルを統合。3 階層対応（COO / lead / member / personal_clone）の土台付き。
処理
召喚 → スキル context 注入 → BMAD ペルソナ選択 → parent 継承 → LLM 抽象化（M-20）経由呼び出し → artifact 保存 + AC 検証（M-25）
権限
run_session

概要
hearing → requirements 連鎖 + 画面ごと 1 HTML モック自動生成 + コンポーネントカタログ + 画面遷移マップ
出力
HTML サマリー / 要件定義書 / 画面モック群 / カタログ / 遷移マップ / Markdown 併記
制約
Tailwind CDN + SVG + 印刷対応 / 1 画面 1MB 以内 / responsive md:(768px+)
M-9依存グラフ + 基本伝搬Phase 1

Evaluator 出力をリーダー AI が壁打ち。3 ターン改善なしで人間エスカレ。

M-12 + M-26赤線リスト + Constitution（プロジェクト不変原則）Phase 1

1. API キー漏洩 / 2. 本番 DB 破壊 / 3. GitHub force push / 4. AI 無限ループ / 5. デプロイ独断

プロジェクト不変原則 Markdown / 全 AI セッションが context として参照 / 違反は赤線エスカレ
