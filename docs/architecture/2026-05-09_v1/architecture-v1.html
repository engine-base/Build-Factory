全体方針
アーキパターン

モジュラーモノリス（FastAPI + Next.js）確定

選定理由
  • 1 人開発（dogfooding）+ Claude Code 実装エンジンで運用負荷を最小化
  • swarm 50 並列セッション + WebSocket は単一プロセスが状態同期しやすい
  • Supabase が DB / Auth / Realtime / Storage を一手に担うため Backend を軽量化
  • 13 モジュール境界を明確化することで Phase 2/3 で必要箇所のみマイクロサービス化可能
採用しなかった選択肢
選択肢採用しない理由
モノリス（無構造）Phase 1.5 以降で機能間の境界が曖昧化し分離困難
マイクロサービス1 人運用で K8s / service mesh / gRPC は持続不可・over-engineering
サーバレス全面subprocess + 長時間 swarm + WebSocket は Lambda 不向き

DB 設計方針（要点）
設計原則
  • PK：UUID v7（時系列ソート可能）
  • 削除：ソフトデリート（deleted_at）+ RLS で auto フィルタ
  • マルチテナント：account_id / workspace_id 全テーブル必須 + RLS で auto enforce
  • 命名：snake_case / 複数形 / FK は <table>_id
  • タイムスタンプ：created_at / updated_at（trigger 自動更新）
  • 監査：主要テーブルに変更検出 trigger → audit_logs に自動 INSERT
  • 時系列：BRIN index + pg_partman 月次 partition（P2）

領域対策
認証Supabase Auth JWT + 2FA + OAuth（Anthropic / Slack / GitHub）
認可Supabase RLS + custom_permissions JSONB / OpenFGA（P2 補完）
通信HTTPS + wss:// + CORS ホワイトリスト + CSP + CSRF（SameSite=Lax）
DB at restSupabase disk encryption + pgsodium で API キー / トークン暗号化
クライアント側 JWThttpOnly + Secure + SameSite cookie
NDA 案件機密workspace 機密ラベル + AI 取得時フィルタ + 横断検索除外
個人クローンopt-in 必須（DEFAULT FALSE）+ namespace 完全分離 + GDPR ready
レートリミットAPI 1000/5min/user / Cloudflare 10000/5min/IP / LLM コスト workspace 上限
秘匿情報Coolify Secrets / GitHub Actions Secrets / .env / pre-commit gitleaks

赤線リスト 5 項目（自動防御）
  1. 顧客 API キー / 秘密情報の漏洩
  2. 本番 DB への破壊的書込み（DROP TABLE / DELETE 全件 / TRUNCATE）
  3. GitHub force push / 他人ブランチ書換
  4. AI コール無限ループ（コスト爆発）
  5. デプロイ判断の独断

設計トレードオフ一覧
採用犠牲になるもの受容理由
モジュラーモノリス1 サービス障害で全停止1 人運用なら許容・冗長化は P3
Supabase 採用中程度のベンダーロックイン標準 SQL + RLS portable で緩和 / 統合性のメリット大
LiteLLM self-hostVPS 運用負荷Langfuse 統合と OSS / コストメリット大
自前 WebSocket（swarm）開発コスト + sticky sessionswarm の 100ms 要件は Supabase Realtime 不向き
Vercel + Oracle Free Tier環境分散初期コスト ¥0 / dogfood に十分
pgsodiumSupabase Vault GA で再検討現時点での標準 + Postgres ネイティブ
GitHub ActionsGitHub 依存（P3 で問題なら GitLab 移行可）M-13 GitHub 連携と統合性高
個人クローン opt-in OFF デフォルト学習データ集まりにくいプライバシー保守 / 信頼獲得
Supabase CLI（alembic 廃止）既存資産の一部破棄DB 統一性 / RLS 管理性
