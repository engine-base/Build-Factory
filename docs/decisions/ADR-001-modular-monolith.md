# ADR-001: モジュラーモノリス採用 (vs マイクロサービス)

- **Status**: Accepted
- **Date**: 2026-05-09
- **Deciders**: 高本まさと

## Context

Build-Factory のバックエンドを設計するにあたり、3 つの選択肢があった:

1. **マイクロサービス** = 機能ごとに独立サービス + メッセージング
2. **モジュラーモノリス** = 単一プロセス、ドメインごとにモジュール分離
3. **シンプルモノリス** = 全部 1 つ

主要な制約:
- **1 人 (masato) で運用** = 運用コスト最小化が必須
- **Phase 1 は ¥0/月** = 複数サービスを動かす予算がない
- **AI 社員が実装する** = 境界が明確な構造である必要がある (LLM が混乱しない)
- **Phase 2 で SaaS 化** = 将来のスケールも考慮

## Decision

**モジュラーモノリス (FastAPI + 13+ ドメインモジュール)** を採用する。

```
backend/
├── routers/        # 40 ルーター (HTTP エンドポイント)
├── services/       # 50 サービス (ビジネスロジック)
├── domains/
│   ├── auth/       # 独立性の強いドメイン
│   ├── workspace/
│   ├── hearing/
│   ├── spec/
│   ├── task/
│   ├── execution/
│   ├── review/
│   ├── ai_employee/
│   ├── memory/
│   └── ...
└── main.py
```

各ドメインは:
- 自身のモデル / サービス / ルーターを持つ
- 他ドメインへは **公開インターフェース経由のみ** で呼び出す (依存方向を制御)
- ドメイン間の循環依存を禁止

## Consequences

### 得られるもの
- ✅ Phase 1 でデプロイ 1 つ、運用 1 つ → ¥0 達成
- ✅ AI 社員 (devon) が「○○ ドメインを修正して」で作業範囲を明確化
- ✅ Phase 2 で SaaS 公開時、必要なドメインだけマイクロサービス分離可能 (既に境界が明確なので)
- ✅ bootstrap 済みの 40 routers + 50 services をそのまま REUSE 可能

### 諦めるもの
- ❌ 単一プロセスなので、特定ドメインのスケール課題は全体に波及
  - → Phase 2 で対象ドメインを切り出す形で対応
- ❌ ドメイン間で言語を分けられない (全部 Python)
  - → 現状デメリットなし、Phase 2 で必要なら gRPC / sidecar で対応

### 関連
- 影響を受けるタスク: T-001-01 (FastAPI 基盤) / T-001-04 (middleware) / T-S0-09 (sandbox)
- 参考: Shopify の Modular Monolith 事例 / DDD bounded context
